## 1. 基础约定

二轮充电是独立模块，接口使用 `/twoWheel/*`、小程序接口使用 `/twoWheel/xcx/*`、支付接口使用 `/pay/twoWheel*`。后端实现可以继承四轮 `/pay/charging` 的成熟支付流程，但二轮前端和小程序正式对接新接口名；二轮业务主数据只写 `two_wheel_*`，不写四轮 `charging_*` 业务表。

`BKV` 只是可选设备子品牌。当桩或枪口 `device_brand='BKV'` 时，启动和停止会调用 `barrier_gate_system` 的 BKV 中间件；创建站、桩、枪时不默认强制为 BKV。

通用成功响应：

```json
{
  "success": true,
  "data": {}
}
```

通用失败响应：

```json
{
  "success": false,
  "error": "错误原因"
}
```

## 2. 枚举和公共字段

### 2.1 站点状态 status

| 值 | 说明 | 前端展示 |
| --- | --- | --- |
| enable | 正常收费 | 可正常扫码、下单、充电 |
| repair | 施工维护 | 站点维护中 |
| disable | 关闭 | 站点关闭 |

当前代码已对站点创建、修改、状态变更校验以上三个值。

### 2.2 三种收费模式

| 字段 | 模式值 | 说明 |
| --- | --- | --- |
| duration_strategy_id | duration | 按时长收费 |
| electricity_service_strategy_id | electricity_service | 电价服务费分离 |
| per_use_strategy_id | per_use | 按次收费 |

站点表最终写入字段为 `two_wheel_strategy_ids`，格式如下：

```json
{
  "duration": "STRATEGY_ID_1",
  "electricity_service": "STRATEGY_ID_2",
  "per_use": "STRATEGY_ID_3"
}
```

创建或修改站点时，可以直接传 `two_wheel_strategy_ids`，也可以传上面的三个友好字段。代码会自动归一化。

收费模式业务流程：

1. 后台先通过 `/twoWheel/strategy/create` 创建三类策略，`type` 分别为 `duration`、`electricity_service`、`per_use`。
2. 后台创建或修改站点时，把策略 ID 绑定到站点。可以传 `duration_strategy_id`、`electricity_service_strategy_id`、`per_use_strategy_id`，也可以直接传 `two_wheel_strategy_ids`。
3. 小程序扫码后展示站点可用收费模式。可用模式以站点 `two_wheel_strategy_ids` 中存在的 key 为准。
4. 小程序创建订单或支付充电订单时，传 `charging_mode` 指定本次使用的收费模式。可选值为 `duration/electricity_service/per_use`。如前端已经明确策略，也可以同时传 `strategy_id`；后端应校验该策略属于当前站点对应模式。

创建站点时的策略绑定示例：

```json
{
  "station_no": "TWS001",
  "name": "东门二轮充电站",
  "status": "enable",
  "duration_strategy_id": "STRATEGY_DURATION_ID",
  "electricity_service_strategy_id": "STRATEGY_ES_ID",
  "per_use_strategy_id": "STRATEGY_PER_USE_ID"
}
```

小程序下单或支付时的模式选择示例：

```json
{
  "two_wheel_station_id": "STATION_ID",
  "two_wheel_port_id": "PORT_ID",
  "charging_mode": "duration",
  "strategy_id": "STRATEGY_DURATION_ID"
}
```

### 2.3 白名单开关

站点字段 `whitelist_apply` 表示是否启用白名单免支付策略：

| 值 | 说明 |
| --- | --- |
| true | 当前站点参与白名单免支付判断 |
| false | 当前站点不使用白名单免支付 |

白名单真正免支付需要同时满足：站点 `whitelist_apply=true`、用户或手机号命中白名单、白名单 `status=enable`、未过期、站点适用。

## 4. 后台提交接口

### POST /twoWheel/station/create

创建二轮充电站。

#### 四轮旧接口对照和二轮继承说明

| 对照项 | 原四轮充电站创建 | 二轮充电站创建 |
| --- | --- | --- |
| 前端接口 | `POST /charging/station/new` | `POST /twoWheel/station/create` |
| 数据落点 | `charging_station` | `two_wheel_station` |

二轮对旧设计的继承和增改：

| 类型 | 内容 |
| --- | --- |
| 继承 | 仍然按“站点主数据”创建，保留站点编号、名称、归属、地址、省市区、坐标、标签、图片、运营说明、发票、白名单开关、扩展 `options` 等旧模型能力。 |
| 继承 | 当前实现已经按四轮站点创建口径补齐：仍校验 `name/data_owner_id/address/province/city/district/status`，状态仍只能为 `enable/repair/disable`。 |
| 继承 | 仍然为后续桩、枪、订单和计费策略提供站点主键；前端创建站点后，应使用返回的 `data.id` 作为二轮桩和枪的 `two_wheel_station_id`。 |
| 继承 | 旧四轮 `charging_pile_num/charging_port_num` 的统计口径保留，但二轮实际数量以后续 `two_wheel_pile/two_wheel_port` 为准，可由后台统计回填。 |
| 继承 | 状态继续沿用四轮含义和枚举：`enable` 正常收费，`repair` 施工维护，`disable` 关闭。 |
| 修改 | 接口名从 `/charging/station/new` 换为 `/twoWheel/station/create`，二轮站点创建以新接口作为正式入口。 |
| 修改 | 数据不再写 `charging_station`，只写 `two_wheel_station`；后续二轮订单、支付、BKV 回调也只关联 `two_wheel_*`。 |
| 修改 | `charging_station_no` 拆换为 `station_no/two_wheel_station_no`：`station_no` 是二轮业务编号，`two_wheel_station_no` 是外部/设备侧兼容编号；不传时默认等于 `station_no`。 |
| 修改 | 四轮单策略 `charging_strategy_id` 扩展为二轮多策略 `two_wheel_strategy_ids`，支持按时长、度电+服务费、按次三种二轮计费模式。 |
| 修改 | 前端传入的 `options` 会被保留，并默认补 `logic_status=enable`。 |
| 新增 | 增加 `business_hours/remark/tags` 等更适合小程序展示的字段。 |

字段映射：

| 四轮旧字段 | 二轮字段 | 说明 |
| --- | --- | --- |
| `charging_station_no` | `station_no` | 二轮站点业务编号；和 `two_wheel_station_no` 至少传一个。 |
| `charging_station_no` | `two_wheel_station_no` | 二轮外部编号，选填；不传默认等于 `station_no`。 |
| `name` | `name` | 沿用站点名称。 |
| `data_owner_id` | `data_owner_id` | 沿用数据归属。 |
| `address/province/city/district/longitude/latitude` | 同名字段 | 沿用地址和坐标。 |
| `status=enable` | `status=enable` | 正常收费/可用，沿用四轮含义。 |
| `status=repair` | `status=repair` | 施工维护，沿用四轮含义。 |
| `status=disable` | `status=disable` | 关闭，沿用四轮含义。 |
| `tag/infrastructure_tags/images` | `tag/infrastructure_tags/images/tags` | 保留旧标签和图片，同时新增 `tags` 便于小程序展示。 |
| `charging_strategy_id` | `duration_strategy_id/electricity_service_strategy_id/per_use_strategy_id` 或 `two_wheel_strategy_ids` | 从四轮单策略变为二轮多策略映射。 |
| `whitelist_apply` | `whitelist_apply` | 继续表示是否启用白名单。 |
| `operating_description/comment/options` | `operating_description/comment/options/remark` | 保留扩展说明，新增 `remark` 作为前端备注。 |

请求参数：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| station_no | string | 条件必填 | 站点编号；和 `two_wheel_station_no` 至少传一个 |
| name | string | 是 | 站点名称 |
| two_wheel_station_no | string | 条件必填 | 二轮外部编号；和 `station_no` 至少传一个，不传时默认等于 `station_no` |
| data_owner_id | string | 是 | 数据归属，沿用四轮必填规则 |
| address | string | 是 | 参考地址/站点地址 |
| province | string | 是 | 省 |
| city | string | 是 | 市 |
| district | string | 是 | 区县 |
| longitude | number | 否 | 经度 |
| latitude | number | 否 | 纬度 |
| status | string | 是 | 运营/使用状态：`enable` 正常收费，`repair` 施工维护，`disable` 关闭 |
| tag | array | 否 | 站点标签 |
| tags | array | 否 | 站点标签，和 `tag` 均可用于展示 |
| whitelist_apply | boolean | 否 | 是否启用白名单策略；默认 `false` |
| duration_strategy_id | string | 否 | 按时长策略 ID |
| electricity_service_strategy_id | string | 否 | 电价分离策略 ID |
| per_use_strategy_id | string | 否 | 按次策略 ID |
| two_wheel_strategy_ids | object | 否 | 三种策略映射；可替代上面三个策略字段 |
| charging_pile_num | number | 否 | 充电桩数量；创建后可由桩表统计回填 |
| charging_port_num | number | 否 | 插孔数量；创建后可由枪口表统计回填 |
| business_hours | string | 否 | 营业时间 |
| images | array | 否 | 站点图片 |
| operating_description | string | 否 | 运营说明 |
| remark | string | 否 | 备注 |
| comment | string | 否 | 说明 |
| options | object | 否 | 扩展字段 |

请求示例：

```json
{
  "station_no": "TWS001",
  "name": "东门二轮充电站",
  "data_owner_id": "DATA_OWNER_ID",
  "address": "园区东门",
  "longitude": 119.30000000,

  "province": "福建省",
```

| 对照项 | 原四轮充电站修改 | 二轮充电站修改 |

| 前端接口 | `POST /charging/station/edit` | `POST /twoWheel/station/update` |
| 数据落点 | `charging_station` | `two_wheel_station` |

二轮对旧设计的继承和增改：

| 类型 | 内容 |
| --- | --- |
| 继承 | 沿用四轮“按站点主键修改站点主数据”的设计，`id` 为必填，其他字段按需传入。 |
| 继承 | 状态仍使用四轮枚举 `enable/repair/disable`，含义不变。 |

| 修改 | 接口名从 `/charging/station/edit` 换为 `/twoWheel/station/update`。 |
| 修改 | 数据不再写 `charging_station`，只更新 `two_wheel_station`。 |
| 修改 | 原四轮 `charging_strategy_id` 改为二轮 `two_wheel_strategy_ids` 多策略映射，也可以继续用三个友好字段更新。 |

| data_owner_id | string | 否 | 数据归属 |
| parking_lot_id | string | 否 | 关联停车场；下属枪口已绑定车位时不允许修改 |

| whitelist_apply | boolean | 否 | 是否启用白名单策略 |
| duration_strategy_id | string | 否 | 按时长策略 ID |
| electricity_service_strategy_id | string | 否 | 电价分离策略 ID |
| per_use_strategy_id | string | 否 | 按次策略 ID |
| two_wheel_strategy_ids | object | 否 | 三种策略映射 |
| address/longitude/latitude | string/number | 否 | 地址坐标 |
| province/city/district | string | 否 | 省市区 |
| tag/tags | array | 否 | 站点标签 |
| charging_pile_num | number | 否 | 充电桩数量 |

### POST /twoWheel/station/status

修改站点状态。兼容单站点 `id + status`，也兼容四轮旧接口的批量 `ids + newValue`。

#### 四轮旧接口对照和二轮继承说明

| 对照项 | 原四轮充电站状态修改 | 二轮充电站状态修改 |
| --- | --- | --- |
| 前端接口 | `POST /charging/station/updateStatusMany` | `POST /twoWheel/station/status` |
| 数据落点 | `charging_station` | `two_wheel_station` |

二轮对旧设计的继承和增改：

| 类型 | 内容 |
| --- | --- |
| 继承 | 沿用四轮站点状态含义：`enable` 正常收费，`repair` 施工维护，`disable` 关闭。 |
| 继承 | 沿用四轮旧接口批量修改能力，支持 `ids + newValue`。 |
| 继承 | 当前实现会校验状态值，并修改 `status/status_updated_at`。 |
| 修改 | 二轮同时保留单站点 `id + status` 写法；`newValue/status/station_status` 都可作为状态字段。 |
| 修改 | 数据不再写 `charging_station`，只更新 `two_wheel_station.status/status_updated_at`。 |
| 修改 | 操作日志逐站写入 `base_log`，类型为 `update_two_wheel_station_status`。 |

请求参数：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| id | string | 与 `ids` 二选一 | 单个站点主键 |
| ids | array/string | 与 `id` 二选一 | 批量站点主键数组；也兼容逗号分隔字符串 |
| status | string | 与 `newValue/station_status` 三选一 | `enable/repair/disable` |
| newValue | string | 与 `status/station_status` 三选一 | 兼容四轮旧接口状态字段，`enable/repair/disable` |
| station_status | string | 与 `status/newValue` 三选一 | 兼容字段，`enable/repair/disable` |

请求示例：

```json
{
  "id": "STATION_ID",
  "status": "disable"
}
```

批量请求示例：

```json
{
  "ids": ["STATION_ID_1", "STATION_ID_2"],
  "newValue": "disable"
}
```

响应示例：

```json
{
  "success": true,
  "data": "修改成功"
}
```

批量响应示例：

## 5. 小程序支付流程

二轮小程序业务入口摘要：

| 场景 | 接口 | 请求关键字段 | 响应关键字段 | 说明 |
| --- | --- | --- | --- | --- |
| 扫码信息 | `POST /twoWheel/xcx/scanInfo` | `pile_no/device_no/two_wheel_port_id`、`port_no`、`user_id/mobile` | `data.station`、`data.pile`、`data.port`、`data.benefits`、`data.protocol_hint` | 获取站、桩、枪和可用权益。 |
| 创建普通充电订单 | `POST /twoWheel/xcx/order/create` | `user_id`、设备定位字段、`billing_mode/charge_mode`、可选 `strategy_id`、`identity_source=user` | `data.id`、`data.order_no`、`data.status`、`data.pay_status`、金额字段 | 创建 `two_wheel_order`。 |
| 普通充电订单支付 | `POST /pay/twoWheelCharging` | `two_wheel_order_id/two_wheel_order_ids`、`method`、`pay_type`、`identity_source=user` | `trade_id`、`trade_no`、`total_price`、`data.internal_trades` | 创建支付流水，后续支付回调完成后自动启充。 |
| 月卡充电 | `POST /pay/twoWheelCharging` | `order_id`、`identity_source=monthly_card`、`month_card_id`、`user_id` | `data.identity_source`、`data.pay_free`、`data.rows`、`data.card` | 校验月卡后直接启充，不创建普通支付流水。 |
| 充值卡充电 | `POST /pay/twoWheelCharging` | `order_id`、`identity_source=charging_card/recharge_card`、`recharge_card_id`、`user_id` | `data.identity_source`、`data.pay_free`、`data.rows`、`data.card` | 校验充值卡后直接启充，不创建普通支付流水。 |
| 白名单充电 | `POST /pay/twoWheelCharging` | `order_id`、`identity_source=whitelist`、`user_id/mobile` | `data.identity_source`、`data.pay_free`、`data.rows` | 命中白名单后直接启充，不创建普通支付流水。 |
| 购买/续费月卡 | `POST /pay/twoWheelMonthCard` | `month_card_id/month_card_ids`、`method`、`pay_type`、`user_id` | `trade_id`、`trade_no`、`total_price`、`data.internal_trades` | 独立月卡购买/续费业务。 |
| 充值卡充值 | `POST /pay/twoWheelChargingCard` | `recharge_card_id`、`amount`、`method`、`pay_type`、`user_id` | `trade_id`、`trade_no`、`total_price`、`data.internal_trades` | 独立充值卡充值业务。 |
| 查询订单状态 | `POST /twoWheel/xcx/order/status` | `order_id/id` | `data.status`、`data.pay_status`、`data.billing_status`、金额和快照字段 | 查询支付、启充、结算后的订单状态。 |
| 停止充电 | `POST /twoWheel/xcx/stopCharge` | `order_id`、可选 `stop_reason` | `data`、`send_to_device`、`device_response` | BKV 正常停充后等待后台交易记录回调结算。 |

`POST /twoWheel/xcx/startCharge` 定位为手动启充、设备下发测试和异常兜底接口。普通支付成功后由后端在 `finishTwoWheelTrade` 中自动触发启充；月卡、充值卡、白名单链路由 `/pay/twoWheelCharging` 的权益分支触发启充。

四轮旧支付来源：`/pay/charging`、`payForChargingOrder`、`finishTrade`、`finishChargingTrade`。

二轮当前支付实现继承四轮支付框架：创建 `out_trade/internal_trade`，支付渠道回调后由 `service.pay.finishTrade` 统一完成外部流水，再按内部流水 `trade_type` 分流到 `service.twoWheel.finishTwoWheelTrade`。差异是业务落点改为 `two_wheel_order`、`two_wheel_month_card`、`two_wheel_recharge_card`，不再单独创建二轮专用支付表。

小程序业务入口对照：

| 场景 | 二轮前端接口 | 四轮旧接口/旧流程来源 | 内部流水类型 | 数据落点 |
| --- | --- | --- | --- | --- |
| 普通二轮充电订单支付 | `/pay/twoWheelCharging` | `/pay/charging`、`payForChargingOrder`、`finishTrade`、`finishChargingTrade` | `two_wheel_charging`，后付兼容 `two_wheel_charging_after` | `out_trade`、`internal_trade`、`two_wheel_order` |
| 购买/续费二轮月卡 | `/pay/twoWheelMonthCard` | 四轮支付框架、`finishTrade`、旧 `timeCard` 购买激活流程 | `pay_two_wheel_month_card` | `out_trade`、`internal_trade`、`two_wheel_month_card`、`two_wheel_month_card_log` |
| 充值二轮充值卡 | `/pay/twoWheelChargingCard` | 四轮支付框架、旧 `chargingCard/handle` 充值流程 | `pay_two_wheel_charging_card` | `out_trade`、`internal_trade`、`two_wheel_recharge_card`、`two_wheel_recharge_card_log` |

普通二轮充电订单支付流程：

| 步骤 | 原四轮充电订单支付流程 | 当前二轮充电订单支付流程 |
| --- | --- | --- |
| 1. 创建订单 | `POST /charging/order/create` 路由到 `controller.chargingOrder.create`，创建 `charging_order`。 | `POST /twoWheel/xcx/order/create` 路由到 `controller.xcxTwoWheel.createOrder -> service.twoWheel.createOrder`，创建 `two_wheel_order`。 |
| 2. 发起支付 | `POST /pay/charging` 接收 `charging_order_id/order_ids`。 | `POST /pay/twoWheelCharging` 接收 `two_wheel_order_id/two_wheel_order_ids`，后端实现继承四轮支付流程。 |
| 3. 创建支付流水 | `controller.pay.payForChargingOrder` 创建 `out_trade/internal_trade`，内部流水类型为 `charging_pre/charging_after` 等。 | `controller.pay.payForTwoWheelCharging -> service.twoWheel.payTwoWheelCharging` 创建 `out_trade/internal_trade`，内部流水类型为 `two_wheel_charging`，后付兼容 `two_wheel_charging_after`。 |
| 4. 调用支付网关 | `POST /pay/charging` 返回 `trade_id/trade_no/total_price`，用于后续微信、支付宝等支付网关调用。 | `POST /pay/twoWheelCharging` 返回结构沿用四轮支付习惯，仍包含 `trade_id/trade_no/total_price`。 |
| 5. 支付回调 | 支付渠道回调进入原支付通知，`service.pay.finishTrade` 先完成 `out_trade`，再按内部流水分流。 | 支付渠道回调仍进入原支付通知，继续复用 `service.pay.finishTrade` 完成 `out_trade` 和金额拆分。 |
| 6. 完成业务流水 | 四轮 `charging_pre/charging_after` 调用 `finishChargingTrade`，更新 `charging_order` 的已付金额、预付/补缴字段。 | 二轮 `two_wheel_charging/two_wheel_charging_after` 调用 `service.twoWheel.finishTwoWheelTrade`，更新 `two_wheel_order` 的已付金额、预付/补缴字段。 |
| 7. 支付后动作 | 四轮支付成功后按旧充电逻辑通知设备或等待后续状态。 | 二轮普通支付成功后自动调用二轮启充逻辑，BKV 设备下发结果写入 `two_wheel_order.snapshot`。 |
| 8. 设备结算 | 四轮设备交易记录回调，如 YKC/TLD/HHD/JCZT 的 `tradeRecord`，或 BKV `/bkv/tradeRecord` 未命中二轮订单时，结算 `charging_order`。 | `POST /bkv/tradeRecord` 是设备/中间件侧后台回调，不由小程序调用；后端先调用 `service.twoWheelBkv.findOrderFromCallback` 按二轮订单分流，命中后由 `service.twoWheelBkv.finishFromTradeRecord` 结算 `two_wheel_order`；未命中才回落四轮 `charging_order`。 |

继承和改动总结：

| 类型 | 内容 |
| --- | --- |
| 继承 | 保留四轮支付的 `out_trade/internal_trade`、支付渠道回调、金额拆分、幂等完成流程。 |
| 继承 | 支付接口返回继续使用 `trade_id/trade_no/total_price`，响应结构保持原支付网关所需字段。 |
| 修改 | 二轮充电订单支付接口名换为 `/pay/twoWheelCharging`，`/pay/charging` 作为四轮旧流程来源保留。 |
| 修改 | 业务订单落点从 `charging_order` 改为 `two_wheel_order`；支付流水沿用 `out_trade/internal_trade`，不新增二轮专用支付表。 |
| 修改 | 内部流水类型从 `charging_pre/charging_after` 等四轮类型改为 `two_wheel_charging/two_wheel_charging_after`。 |
| 新增 | 支付成功后自动进入二轮启充；BKV 回调先查二轮订单，确保新链路不误写四轮订单。 |

权益免支付流程：

| identity_source | 调用接口 | 后端动作 |
| --- | --- | --- |
| `whitelist` | `/pay/twoWheelCharging` | 校验 `two_wheel_whitelist`，订单标记免费后直接启充 |
| `monthly_card` | `/pay/twoWheelCharging` | 校验 `two_wheel_month_card`，写订单身份快照后直接启充 |
| `charging_card` / `recharge_card` | `/pay/twoWheelCharging` | 校验 `two_wheel_recharge_card`，写订单身份快照后直接启充 |

补充说明：

- 普通付费订单入口为 `/pay/twoWheelCharging`；`/pay/charging` 只作为四轮旧支付流程来源。
- 订单字段可用 `order_id/order_ids/two_wheel_order_id/two_wheel_order_ids`，后端统一解析为二轮订单 ID。
- 普通支付接口返回支付流水；启动状态通过 `/twoWheel/xcx/order/status` 查询。
- `/bkv/tradeRecord` 是设备/中间件侧后台回调，不由小程序调用。
- 用户主动停充或设备完成后，最终费用以后台收到 `/bkv/tradeRecord` 并结算后的订单状态为准。

### POST /pay/twoWheelMonthCard

### POST /twoWheel/xcx/stopCharge

#### 四轮旧接口对照和二轮继承说明

| 数据落点 | `charging_order`，权益落点 `time_card/charging_card` | `two_wheel_order`，权益落点 `two_wheel_month_card/two_wheel_recharge_card` |

二轮对旧设计的继承和增改：

| 类型 | 内容 |

### POST /twoWheel/xcx/startCharge

> 重要说明：该接口放在小程序业务接口章节最后，定位为手动启充、设备下发测试和异常兜底接口。普通付费主流程由后端在 `finishTwoWheelTrade` 中自动触发二轮启充；月卡、充值卡、白名单等免支付链路由 `/pay/twoWheelCharging` 的权益分支触发启充。

#### 四轮旧接口对照和二轮继承说明

| 对照项 | 原四轮/BKV 启动充电 | 二轮手动启充/调试启充 |
| --- | --- | --- |
| 前端接口 | `POST /xcx/bkv/startCharge`，设备下发参考 BKV `/bkv/remotecontrolboot` | `POST /twoWheel/xcx/startCharge` |
| 数据落点 | `charging_order` | `two_wheel_order` |

二轮对旧设计的继承和增改：

| 类型 | 内容 |
| --- | --- |
| 继承 | 仍按订单、设备、端口、计费快照生成设备启动下发参数。 |
| 继承 | BKV 设备仍走 `/bkv/remotecontrolboot` 下发启动命令。 |
| 修改 | 普通小程序主流程不再主动调用该接口，支付成功或权益校验通过后由后端自动触发。 |
| 修改 | 启动状态和 BKV 下发结果写入 `two_wheel_order`，不写旧 `charging_order`。 |
| 新增 | 可用于设备联调、支付后启充失败重试、异常兜底。 |

字段映射：

| 四轮旧字段 | 二轮字段 | 说明 |
| --- | --- | --- |
| `charging_order_id` | `order_id`、兼容 `id` | 被启动的二轮订单 ID。 |
| `billing_mode/charge_mode` | `billing_mode` | BKV 下发使用，通常来自订单快照。 |
| `charge_time_min/max_charge_time_min` | 同名字段 | BKV 下发充电时长。 |
| `total_amount_cent` | `total_amount_cent` | BKV 下发金额分，默认按订单金额换算。 |
| BKV 启动 payload | `bkv_payload` | 透传给 BKV 中间件。 |
| `charging_order.snapshot` | `two_wheel_order.snapshot.bkv_start_command` | 记录启动请求和设备响应。 |

| 场景 | 说明 |

| 特殊运营兜底 | 确认订单状态和支付状态正确后，由后台人员手动触发启充。 |

请求参数：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| order_id | string | 是 | 订单 ID |
| billing_mode | string | 否 | `duration/electricity_service/per_use`，BKV 下发使用 |

| total_amount_cent | number | 否 | 金额分，默认订单金额乘 100 |
| bkv_payload | object | 否 | 透传给 BKV 中间件的额外字段 |

响应参数：

## 7. 数据库表说明

二轮充电业务表统一使用 `two_wheel_*` 前缀。BKV 只作为 `device_brand='BKV'` 的设备品牌，不作为业务模块名。

| 表名 | 用途 | 关键字段 | 业务关系 |
| --- | --- | --- | --- |
| `two_wheel_station` | 二轮充电站 | `station_no`、`name`、`address`、`status`、`whitelist_apply`、`two_wheel_strategy_ids`、`charging_pile_num`、`charging_port_num` | 站点下挂多个二轮桩和插孔；通过 `two_wheel_strategy_ids` 绑定三种计费策略 |
| `two_wheel_pile` | 二轮充电桩 | `pile_no`、`device_no`、`device_mac`、`device_name`、`brand_model`、`two_wheel_station_id`、`device_status`、`charging_port_num` | 归属 `two_wheel_station`；一台桩可包含多个 `two_wheel_port` |
| `two_wheel_port` | 二轮充电口/插孔 | `port_no`、`device_no`、`two_wheel_pile_id`、`two_wheel_station_id`、`qrcode_id`、`max_power`、`port_status`、`using_status` | 归属桩和站点；小程序扫码、下单和设备启停都定位到插孔 |
| `two_wheel_strategy` | 二轮计费策略 | `name`、`type`、`price`、`electric_price`、`service_price`、`duration_minutes`、`rules.per_use_options`、`description`、`created_by` | 支持 `duration` 时长收费、`electricity_service` 电价分离、`per_use` 多档位按次收费 |
| `two_wheel_order` | 二轮充电订单 | `order_no`、`user_id`、`two_wheel_station_id`、`two_wheel_pile_id`、`two_wheel_port_id`、`status`、`charging_time`、`finish_time`、`price`、`paid_fee`、`refund_fee`、`trade_no`、`device_order_no` | 关联站、桩、口、策略和支付流水；记录充电生命周期、金额、支付、退款、补缴和设备订单信息 |
| `two_wheel_whitelist` | 二轮白名单 | `mobile`、`station_ids`、`unlimited`、`start_time`、`end_time`、`status`、`remark` | 站点启用 `whitelist_apply=true` 且白名单命中时，可作为免支付权益通道 |
| `two_wheel_month_card_config` | 二轮月卡套餐 | `name`、`original_price`、`price`、`total_time_hours`、`expire_type`、`valid_days`、`expire_at`、`station_ids`、`debt_check_type`、`agreement_type`、`notice_enabled` | 后台配置套餐；用户购买后生成 `two_wheel_month_card` |
| `two_wheel_month_card` | 用户月卡 | `card_no`、`config_id`、`config_name`、`user_id`、`user_name`、`mobile`、`status`、`start_date`、`end_date`、`left_time`、`paid_fee`、`recommender_id`、`add_source` | 记录用户已购买月卡；充电结算可扣减剩余时长 |
| `two_wheel_month_card_log` | 月卡流水 | `card_id`、`order_id`、`pay_order_id`、`type`、`amount`、`before_left_time`、`after_left_time`、`trade_no` | 记录购买、续卡、充电扣时长、退款、后台调整等时长变化；`pay_order_id` 为兼容字段，当前记录 `internal_trade.id` |
| `two_wheel_recharge_card` | 二轮充值卡 | `card_no`、`denomination`、`balance`、`bind_user_id`、`mobile`、`status`、`remark` | 记录充值卡卡面值和余额；可绑定用户并作为充电支付权益 |
| `two_wheel_recharge_card_log` | 充值卡流水 | `card_id`、`order_id`、`pay_order_id`、`type`、`amount`、`before_balance`、`after_balance`、`before_denomination`、`after_denomination` | 记录创建、充值、消费、退回、调整等余额变化；消费记录可通过 `order_id` 关联充电订单；`pay_order_id` 为兼容字段，当前记录 `internal_trade.id` |

所有 `two_wheel_*` 表都应包含 `id`、`created_at`、`updated_at`、`deleted_at`、`deleted`，用于主键、创建/更新时间和软删除。
