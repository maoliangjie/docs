## 1. 基础约定

二轮充电是独立模块，接口使用 `/twoWheel/*`、小程序接口使用 `/twoWheel/xcx/*`、支付接口使用 `/pay/twoWheel*`。后端实现可以继承四轮 `/pay/charging` 的成熟支付流程，但二轮前端和小程序正式对接新接口名；二轮业务主数据只写 `two_wheel_*`，不写四轮 `charging_*` 业务表。

`BKV` 只是可选设备子品牌。当桩或枪口 `device_brand='BKV'` 时，启动和停止会调用 `barrier_gate_system` 的 BKV 中间件；创建站、桩、枪时不默认强制为 BKV。

BKV 设备通过 TCP 心跳、设备鉴权或实时状态上报接入平台时，默认按二轮设备处理：首次接入会在 `two_wheel_pile` 创建库存桩，并按 `route_count/port_count` 或实时上报的 `port_no/plug_num` 在 `two_wheel_port` 创建插孔；后续心跳和状态上报刷新二轮桩/枪口在线、使用状态和 BKV payload。只有请求明确标识为四轮或 legacy 接入时，才进入旧 `charging_pile/charging_port` 处理路径。

### BKV 设备和订单状态口径

二轮设备状态分为站点、桩、枪口和订单状态。BKV 实时上报会刷新桩/枪口在线和使用状态；订单状态通过 `/twoWheel/xcx/order/status` 展示。

当前二轮设备和订单状态口径：

| 状态字段 | 可见值 | 含义 |
| --- | --- | --- |
| `two_wheel_station.status` | `enable/repair/disable` | 正常收费、施工维护、关闭。 |
| `two_wheel_pile.device_status` | `online/offline`，也兼容展示 `fault/configuring` | 桩在线状态。 |
| `two_wheel_pile.pile_status` | `enable/disable` | 桩启停配置。 |
| `two_wheel_pile.using_status` | `using/stock` | 已投放、库存；桩离线由 `device_status=offline` 表达。 |
| `two_wheel_port.device_status` | `online/offline`，也兼容展示 `fault/configuring` | 枪口在线状态。 |
| `two_wheel_port.port_status` | `enable/disable` | 枪口启停配置。 |
| `two_wheel_port.using_status` | `free/occupy/charging/repair/disconnect/disable/fault` | 空闲、占用、充电中、维修、离线、停用、故障。BKV 实时状态上报会按设备 payload 刷新该字段。 |
| `two_wheel_order.status` | `waiting/charging/finish` | 待启充或待支付、充电中、已结束。 |
| `two_wheel_order.pay_status` | `uncleared/paid/free` | 未结清、已支付、权益免支付。 |
| `two_wheel_order.billing_status` | `waiting/charging/finish` | 待计费、计费中、计费结束。 |

BKV 实时状态进入日志和实时监控时，常用映射为：

| `using_status` | 数字状态 | 说明 |
| --- | --- | --- |
| `free` | `1` | 空闲 |
| `occupy` | `2` | 占用/插入 |
| `charging` | `3` | 充电中 |
| `repair` | `4` | 维修/故障 |
| `disconnect` | `5` | 离线 |

通用成功响应：

```json
{
  "success": true,
  "data": {}
}
```

通用失败响应：

```json
{
  "success": false,
  "error": "错误原因"
}
```

## 2. 枚举和公共字段

### 2.1 站点状态 status

| 值 | 说明 | 前端展示 |
| --- | --- | --- |
| enable | 正常收费 | 可正常扫码、下单、充电 |
| repair | 施工维护 | 站点维护中 |
| disable | 关闭 | 站点关闭 |

当前代码已对站点创建、修改、状态变更校验以上三个值。

### 2.2 三种收费模式

| 字段 | 模式值 | 说明 |
| --- | --- | --- |
| duration_strategy_id | duration | 按时长收费 |
| electricity_service_strategy_id | electricity_service | 电价服务费分离 |
| per_use_strategy_id | per_use | 按次收费 |

站点表最终写入字段为 `two_wheel_strategy_ids`，格式如下：

```json
{
  "duration": "STRATEGY_ID_1",
  "electricity_service": "STRATEGY_ID_2",
  "per_use": "STRATEGY_ID_3"
}
```

创建或修改站点时，可以直接传 `two_wheel_strategy_ids`，也可以传上面的三个友好字段。代码会自动归一化。

收费模式业务流程：

1. 后台先通过 `/twoWheel/strategy/create` 创建三类策略，`type` 分别为 `duration`、`electricity_service`、`per_use`。
2. 后台创建或修改站点时，把策略 ID 绑定到站点。可以传 `duration_strategy_id`、`electricity_service_strategy_id`、`per_use_strategy_id`，也可以直接传 `two_wheel_strategy_ids`。
3. 小程序扫码后展示站点可用收费模式。可用模式以站点 `two_wheel_strategy_ids` 中存在的 key 为准。
4. 小程序创建订单时必须传 `billing_mode/charge_mode` 或 `strategy_id`。传 `billing_mode/charge_mode` 表示按模式选择站点绑定策略；传 `strategy_id` 表示直接指定本次订单使用的计费策略，后端会校验该策略属于当前站点。支付阶段不再决定收费策略。

创建站点时的策略绑定示例：

```json
{
  "station_no": "TWS001",
  "name": "东门二轮充电站",
  "status": "enable",
  "duration_strategy_id": "STRATEGY_DURATION_ID",
  "electricity_service_strategy_id": "STRATEGY_ES_ID",
  "per_use_strategy_id": "STRATEGY_PER_USE_ID"
}
```

小程序下单或支付时的模式选择示例：

```json
{
  "two_wheel_station_id": "STATION_ID",
  "two_wheel_port_id": "PORT_ID",
  "charging_mode": "duration",
  "strategy_id": "STRATEGY_DURATION_ID"
}
```

### 2.3 白名单开关

站点字段 `whitelist_apply` 表示是否启用白名单免支付策略：

| 值 | 说明 |
| --- | --- |
| true | 当前站点参与白名单免支付判断 |
| false | 当前站点不使用白名单免支付 |

白名单真正免支付需要同时满足：站点 `whitelist_apply=true`、用户或手机号命中白名单、白名单 `status=enable`、未过期、站点适用。

## 4. 后台提交接口

### POST /twoWheel/station/create

创建二轮充电站。

#### 四轮旧接口对照和二轮继承说明

| 对照项 | 原四轮充电站创建 | 二轮充电站创建 |
| --- | --- | --- |
| 前端接口 | `POST /charging/station/new` | `POST /twoWheel/station/create` |
| 数据落点 | `charging_station` | `two_wheel_station` |

二轮对旧设计的继承和增改：

| 类型 | 内容 |
| --- | --- |
| 继承 | 仍然按“站点主数据”创建，保留站点编号、名称、归属、地址、省市区、坐标、标签、图片、运营说明、发票、白名单开关、扩展 `options` 等旧模型能力。 |
| 继承 | 当前实现已经按四轮站点创建口径补齐：仍校验 `name/data_owner_id/address/province/city/district/status`，状态仍只能为 `enable/repair/disable`。 |
| 继承 | 仍然为后续桩、枪、订单和计费策略提供站点主键；前端创建站点后，应使用返回的 `data.id` 作为二轮桩和枪的 `two_wheel_station_id`。 |
| 继承 | 旧四轮 `charging_pile_num/charging_port_num` 的统计口径保留，但二轮实际数量以后续 `two_wheel_pile/two_wheel_port` 为准，可由后台统计回填。 |
| 继承 | 状态继续沿用四轮含义和枚举：`enable` 正常收费，`repair` 施工维护，`disable` 关闭。 |
| 修改 | 接口名从 `/charging/station/new` 换为 `/twoWheel/station/create`，二轮站点创建以新接口作为正式入口。 |
| 修改 | 数据不再写 `charging_station`，只写 `two_wheel_station`；后续二轮订单、支付、BKV 回调也只关联 `two_wheel_*`。 |
| 修改 | `charging_station_no` 拆换为 `station_no/two_wheel_station_no`：`station_no` 是二轮业务编号，`two_wheel_station_no` 是外部/设备侧兼容编号；不传时默认等于 `station_no`。 |
| 修改 | 四轮单策略 `charging_strategy_id` 扩展为二轮多策略 `two_wheel_strategy_ids`，支持按时长、度电+服务费、按次三种二轮计费模式。 |
| 修改 | 前端传入的 `options` 会被保留，并默认补 `logic_status=enable`。 |
| 新增 | 增加 `business_hours/remark/tags` 等更适合小程序展示的字段。 |

字段映射：

| 四轮旧字段 | 二轮字段 | 说明 |
| --- | --- | --- |
| `charging_station_no` | `station_no` | 二轮站点业务编号；和 `two_wheel_station_no` 至少传一个。 |
| `charging_station_no` | `two_wheel_station_no` | 二轮外部编号，选填；不传默认等于 `station_no`。 |
| `name` | `name` | 沿用站点名称。 |
| `data_owner_id` | `data_owner_id` | 沿用数据归属。 |
| `address/province/city/district/longitude/latitude` | 同名字段 | 沿用地址和坐标。 |
| `status=enable` | `status=enable` | 正常收费/可用，沿用四轮含义。 |
| `status=repair` | `status=repair` | 施工维护，沿用四轮含义。 |
| `status=disable` | `status=disable` | 关闭，沿用四轮含义。 |
| `tag/infrastructure_tags/images` | `tag/infrastructure_tags/images/tags` | 保留旧标签和图片，同时新增 `tags` 便于小程序展示。 |
| `charging_strategy_id` | `duration_strategy_id/electricity_service_strategy_id/per_use_strategy_id` 或 `two_wheel_strategy_ids` | 从四轮单策略变为二轮多策略映射。 |
| `whitelist_apply` | `whitelist_apply` | 继续表示是否启用白名单。 |
| `operating_description/comment/options` | `operating_description/comment/options/remark` | 保留扩展说明，新增 `remark` 作为前端备注。 |

请求参数：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| station_no | string | 条件必填 | 站点编号；和 `two_wheel_station_no` 至少传一个 |
| name | string | 是 | 站点名称 |
| two_wheel_station_no | string | 条件必填 | 二轮外部编号；和 `station_no` 至少传一个，不传时默认等于 `station_no` |
| data_owner_id | string | 是 | 数据归属，沿用四轮必填规则 |
| address | string | 是 | 参考地址/站点地址 |
| province | string | 是 | 省 |
| city | string | 是 | 市 |
| district | string | 是 | 区县 |
| longitude | number | 否 | 经度 |
| latitude | number | 否 | 纬度 |
| status | string | 是 | 运营/使用状态：`enable` 正常收费，`repair` 施工维护，`disable` 关闭 |
| tag | array | 否 | 站点标签 |
| tags | array | 否 | 站点标签，和 `tag` 均可用于展示 |
| whitelist_apply | boolean | 否 | 是否启用白名单策略；默认 `false` |
| duration_strategy_id | string | 否 | 按时长策略 ID |
| electricity_service_strategy_id | string | 否 | 电价分离策略 ID |
| per_use_strategy_id | string | 否 | 按次策略 ID |
| two_wheel_strategy_ids | object | 否 | 三种策略映射；可替代上面三个策略字段 |
| charging_pile_num | number | 否 | 充电桩数量；创建后可由桩表统计回填 |
| charging_port_num | number | 否 | 插孔数量；创建后可由枪口表统计回填 |
| business_hours | string | 否 | 营业时间 |
| images | array | 否 | 站点图片 |
| operating_description | string | 否 | 运营说明 |
| remark | string | 否 | 备注 |
| comment | string | 否 | 说明 |
| options | object | 否 | 扩展字段 |

### POST /twoWheel/station/update

修改二轮充电站。`update` 可以修改任意支持字段，不限于改名。

#### 四轮旧接口对照和二轮继承说明

| 对照项 | 原四轮充电站修改 | 二轮充电站修改 |
| --- | --- | --- |
| 前端接口 | `POST /charging/station/edit` | `POST /twoWheel/station/update` |
| 数据落点 | `charging_station` | `two_wheel_station` |

二轮对旧设计的继承和增改：

| 类型 | 内容 |
| --- | --- |
| 继承 | 沿用四轮“按站点主键修改站点主数据”的设计，`id` 为必填，其他字段按需传入。 |
| 继承 | 状态仍使用四轮枚举 `enable/repair/disable`，含义不变。 |
| 继承 | 地址、坐标、省市区、标签、图片、运营说明、发票、白名单开关、扩展 `options` 等站点属性继续可修改。 |
| 继承 | 当前实现已对齐四轮修改保护：站点改名会校验同名；修改车场时若下属枪口已绑定车位则拒绝。 |
| 修改 | 接口名从 `/charging/station/edit` 换为 `/twoWheel/station/update`。 |
| 修改 | 数据不再写 `charging_station`，只更新 `two_wheel_station`。 |
| 修改 | 原四轮 `charging_strategy_id` 改为二轮 `two_wheel_strategy_ids` 多策略映射，也可以继续用三个友好字段更新。 |
| 修改 | 站点车场变更会同步更新下属 `two_wheel_pile.parking_lot_id`；这是对四轮“同步桩车场”的二轮表适配。 |
| 新增 | 可修改 `device_brand/service_phone/business_hours/remark/tags` 等二轮小程序展示字段。 |

请求参数：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| id | string | 是 | 站点主键 |
| station_no | string | 否 | 站点编号 |
| two_wheel_station_no | string | 否 | 二轮外部编号 |
| name | string | 否 | 站点名称 |
| data_owner_id | string | 否 | 数据归属 |

`POST /twoWheel/xcx/startCharge` 定位为手动启充、设备下发测试和异常兜底接口。普通支付成功后由后端在 `finishTwoWheelTrade` 中自动触发启充；补缴情形只做欠费结算，不再触发启充；月卡、白名单链路由 `/pay/twoWheelCharging` 的权益分支触发启充；充值卡走 `method=recharge_card` 内部支付路径，扣款成功后触发启充。

四轮旧支付来源：`/pay/charging`、`payForChargingOrder`、`finishTrade`、`finishChargingTrade`。

二轮当前支付实现继承四轮支付框架：创建 `out_trade/internal_trade`，支付渠道回调后由 `service.pay.finishTrade` 统一完成外部流水，再按内部流水 `trade_type` 分流到 `service.twoWheel.finishTwoWheelTrade`。差异是业务落点改为 `two_wheel_order`、`two_wheel_month_card`、`two_wheel_recharge_card`，不再单独创建二轮专用支付表。

| 2. 发起支付 | `POST /pay/charging` 接收 `charging_order_id/order_ids`。 | `POST /pay/twoWheelCharging` 接收 `two_wheel_order_id/two_wheel_order_ids`，后端实现继承四轮支付流程。 |

| 4. 调用支付网关 | `POST /pay/charging` 返回 `trade_id/trade_no/total_price`，用于后续微信、支付宝等支付网关调用。 | `POST /pay/twoWheelCharging` 返回结构沿用四轮支付习惯，仍包含 `trade_id/trade_no/total_price`。 |

| 类型 | 内容 |
| --- | --- |
| 继承 | 保留四轮支付的 `out_trade/internal_trade`、支付渠道回调、金额拆分、幂等完成流程。 |
| 继承 | 支付接口返回继续使用 `trade_id/trade_no/total_price`，响应结构保持原支付网关所需字段。 |
| 修改 | 二轮充电订单支付接口名换为 `/pay/twoWheelCharging`，`/pay/charging` 作为四轮旧流程来源保留。 |
| 修改 | 业务订单落点从 `charging_order` 改为 `two_wheel_order`；支付流水沿用 `out_trade/internal_trade`，不新增二轮专用支付表。 |
| 修改 | 内部流水类型从 `charging_pre/charging_after` 等四轮类型改为 `two_wheel_charging/two_wheel_charging_after`。 |
| 新增 | 支付成功后自动进入二轮启充；BKV 回调先查二轮订单，确保新链路不误写四轮订单。 |

权益免支付流程：

| identity_source | 调用接口 | 后端动作 |
| --- | --- | --- |
| `whitelist` | `/pay/twoWheelCharging` | 校验 `two_wheel_whitelist`，订单标记免费后直接启充 |
| `monthly_card` | `/pay/twoWheelCharging` | 校验 `two_wheel_month_card`，写订单身份快照后直接启充 |
| `charging_card` / `recharge_card` | `/pay/twoWheelCharging` | 旧请求兼容入口；后端转为 `method=recharge_card`，订单仍按 `identity_source=user` 支付和启充 |

补充说明：

- 普通付费订单入口为 `/pay/twoWheelCharging`；`/pay/charging` 只作为四轮旧支付流程来源。
- 订单字段可用 `order_id/order_ids/two_wheel_order_id/two_wheel_order_ids`，后端统一解析为二轮订单 ID。
- 普通支付接口返回支付流水；启动状态通过 `/twoWheel/xcx/order/status` 查询。
- `/bkv/tradeRecord` 是设备/中间件侧后台回调，不由小程序调用。
- 用户主动停充或设备完成后，最终费用以后台收到 `/bkv/tradeRecord` 并结算后的订单状态为准。

### POST /pay/twoWheelCharging

二轮充电统一支付入口。支付阶段不再选择计费策略，只读取订单创建时已固化的 `strategy_id/price_type/charging_type/price/snapshot.bkv_charge_profile`。普通充电订单使用 `method` 决定支付方式；充值卡固定传 `method=recharge_card`。`identity_source` 只用于月卡、白名单等权益身份，旧 `identity_source=charging_card/recharge_card` 仍兼容转为 `method=recharge_card`。

支付字段职责：

| 字段 | 作用 |
| --- | --- |
| `method` | 支付方式。外部支付传 `wx_pay/ali_pay` 等；充值卡支付传 `recharge_card`，后端走内部扣卡路径，不查第三方商户配置。 |
| `identity_source` | 权益身份。普通外部支付可不传或传 `user`；月卡传 `monthly_card`；白名单传 `whitelist`。充值卡不再推荐传该字段。 |
| `recharge_card_id/charging_card_id/card_id` | 当 `method=recharge_card` 时必传，用于指定本次扣款的二轮充值卡。 |
| `billing_mode/strategy_id` | 支付接口不使用。计费策略必须在 `/twoWheel/xcx/order/create` 阶段确定。 |

#### 四轮旧接口对照和二轮继承说明

| 对照项 | 原四轮充电订单支付 | 二轮充电订单支付 |
| --- | --- | --- |
| 前端接口 | `POST /pay/charging` | `POST /pay/twoWheelCharging` |
| 数据落点 | `out_trade`、`internal_trade`、`charging_order` | `out_trade`、`internal_trade`、`two_wheel_order` |

二轮对旧设计的继承和增改：

| 类型 | 内容 |
| --- | --- |
| 继承 | 继续复用四轮支付的 `out_trade/internal_trade`、支付渠道回调、金额拆分和幂等完成流程。 |
| 继承 | 返回继续保留 `trade_id/trade_no/total_price`，小程序可沿用原支付网关封装。 |
| 修改 | 前端入口换为 `/pay/twoWheelCharging`，二轮小程序不直接调用 `/pay/charging`。 |
| 修改 | 内部流水类型按订单状态区分为 `two_wheel_charging` / `two_wheel_charging_after`。 |
| 修改 | 支付成功后更新 `internal_trade/two_wheel_order`；仅 `two_wheel_charging` 会自动触发二轮启充。 |
| 新增 | `method=recharge_card` 支持充值卡作为内部虚拟支付方式，扣卡、写流水、回写订单后启充。 |

字段映射：

| 四轮旧字段 | 二轮字段 | 说明 |
| --- | --- | --- |
| `charging_order_id` | `two_wheel_order_id`、兼容 `order_id` | 单个二轮充电订单 ID。 |
| `charging_order_ids/order_ids` | `two_wheel_order_ids/order_ids` | 多个二轮充电订单 ID。 |
| `charging_pre/charging_after` | `two_wheel_charging/two_wheel_charging_after` | 内部流水类型更名。 |
| `method/pay_type/user_id` | `method/pay_type/user_id` | 支付方式、支付场景、支付用户沿用。 |
| 无 | `identity_source` | 权益来源：`user/monthly_card/whitelist`；旧 `charging_card/recharge_card` 仅做兼容入口。 |
| `charging_order.paid_fee/pay_pre_fee/pay_after_fee` | `two_wheel_order.paid_fee/pay_pre_fee/pay_after_fee` | 支付完成后写入二轮订单金额字段。 |

### POST /twoWheel/xcx/monthCard/create

小程序创建二轮月卡待支付记录。调用成功后返回 `card_id`，前端再调用 `/pay/twoWheelMonthCard` 为该月卡发起支付；支付成功后后端在 `finishTwoWheelTrade` 中把月卡状态置为 `enable`。创建成功后后端会调度 `twoWheelMonthCardTimeout`，若 15 分钟后月卡仍为 `waiting` 且没有成功的 `pay_two_wheel_month_card` 流水，则自动置为 `disable`。

| 类型 | 内容 |

## 7. 数据库表说明

二轮充电业务表统一使用 `two_wheel_*` 前缀。BKV 只作为 `device_brand='BKV'` 的设备品牌，不作为业务模块名。

| 表名 | 用途 | 关键字段 | 业务关系 |
| --- | --- | --- | --- |
| `two_wheel_station` | 二轮充电站 | `station_no`、`name`、`address`、`status`、`whitelist_apply`、`two_wheel_strategy_ids`、`charging_pile_num`、`charging_port_num` | 站点下挂多个二轮桩和插孔；通过 `two_wheel_strategy_ids` 绑定三种计费策略 |
| `two_wheel_pile` | 二轮充电桩 | `pile_no`、`device_no`、`device_mac`、`device_name`、`brand_model`、`two_wheel_station_id`、`device_status`、`charging_port_num` | 归属 `two_wheel_station`；一台桩可包含多个 `two_wheel_port` |
| `two_wheel_port` | 二轮充电口/插孔 | `port_no`、`device_no`、`two_wheel_pile_id`、`two_wheel_station_id`、`qrcode_id`、`max_power`、`port_status`、`using_status` | 归属桩和站点；小程序扫码、下单和设备启停都定位到插孔；`max_power` 单位 kW |
| `two_wheel_strategy` | 二轮计费策略 | `name`、`type`、`price`、`electric_price`、`service_price`、`duration_minutes`、`rules.per_use_options`、`description`、`created_by` | 支持 `duration` 时长收费、`electricity_service` 电价分离、`per_use` 多档位按次收费；时长字段单位分钟，金额字段单位元，电价/服务费单位元/度 |
| `two_wheel_order` | 二轮充电订单 | `order_no`、`user_id`、`two_wheel_station_id`、`two_wheel_pile_id`、`two_wheel_port_id`、`status`、`charging_time`、`finish_time`、`charging_duration`、`charging_power`、`price`、`paid_fee`、`refund_fee`、`trade_no`、`device_order_no` | 关联站、桩、口、策略和支付流水；记录充电生命周期、金额、支付、退款、补缴和设备订单信息；`charging_duration` 单位分钟，`charging_power` 单位度/kWh，金额字段单位元 |
| `two_wheel_whitelist` | 二轮白名单 | `mobile`、`station_ids`、`unlimited`、`start_time`、`end_time`、`status`、`remark` | 站点启用 `whitelist_apply=true` 且白名单命中时，可作为免支付权益通道 |
| `two_wheel_month_card_config` | 二轮月卡套餐 | `name`、`original_price`、`price`、`total_time`、`total_time_hours`、`expire_type`、`valid_days`、`expire_at`、`station_ids`、`debt_check_type`、`agreement_type`、`notice_enabled` | 后台配置套餐；用户购买后生成 `two_wheel_month_card`；`total_time` 单位秒，`total_time_hours` 单位小时 |
| `two_wheel_month_card` | 用户月卡 | `card_no`、`config_id`、`config_name`、`user_id`、`user_name`、`mobile`、`status`、`start_date`、`end_date`、`left_time`、`paid_fee`、`recommender_id`、`add_source`、`remark` | 记录用户已购买月卡；充电结算可扣减剩余时长；`left_time` 单位秒；`remark` 保存购买备注 |
| `two_wheel_month_card_log` | 月卡订单使用记录 | `card_id`、`order_id`、`pay_order_id`、`type`、`amount`、`before_left_time`、`after_left_time`、`trade_no` | 只记录充电订单使用月卡结算的扣时长记录，当前主要为 `type=consume`；`amount/before_left_time/after_left_time` 单位秒；购买、结束等操作写 `base_log` 或 `out_trade/internal_trade` |
| `two_wheel_recharge_card` | 二轮充值卡 | `card_no`、`denomination`、`balance`、`bind_user_id`、`mobile`、`status`、`remark` | 记录充值卡卡面值和余额；可绑定用户并作为充电内部支付方式 |
| `two_wheel_recharge_card_log` | 充值卡金额变动记录 | `card_id`、`order_id`、`pay_order_id`、`type`、`amount`、`before_balance`、`after_balance`、`before_denomination`、`after_denomination` | 有金额变动即写入，金额字段单位元；`type=create/recharge/return/consume/refund` 分别表示创建初始金额、充值增加、退回减少、充电消费、消费退款 |

所有 `two_wheel_*` 表都应包含 `id`、`created_at`、`updated_at`、`deleted_at`、`deleted`，用于主键、创建/更新时间和软删除。
